# Jetson Orin NX GPU — Talos Extension Plan

## Goal

Enable NVIDIA GPU compute (CUDA) on `nv1` — a Jetson Orin NX 16GB worker node in the Talos
cluster — so that GPU-accelerated workloads (ollama, whisper, etc.) can be scheduled there.

---

## Architecture Background (why this is non-trivial)

Jetson Orin NX has an **integrated Tegra/Ampere SoC GPU**, not a discrete PCIe device.

| | Discrete GPU (desktop/server) | Jetson Orin NX |
|---|---|---|
| Bus | PCIe endpoint | Tegra platform bus |
| Driver | `nvidia-open-gpu-kernel-modules` | `nvgpu` |
| PCIe presence | Full GPU endpoint (`10de:xxxx`) | Only PCIe root port controllers (`10de:229c`, driver: `pcieport`) |
| Siderolabs extension | `siderolabs/nvidia-open-gpu-kernel-modules-lts` ❌ | None exists yet |

**The Siderolabs `nvidia-open-gpu-kernel-modules-lts` extension will never work on Jetson.**
The companion service `ext-nvidia-persistenced` waits indefinitely for
`/sys/bus/pci/drivers/nvidia` which is never populated because there is no PCIe GPU endpoint.

The correct driver is **[OE4T/linux-nvgpu](https://github.com/OE4T/linux-nvgpu)** (branch
`patches-r36.5`), a community-maintained Tegra GPU driver targeting non-L4T kernels.

## Action Plan

### Phase 1 — Compilation viability test ✅ COMPLETE

Kubernetes Job `cluster/.tools/nvgpu-build-test.yaml` compiled `OE4T/linux-nvgpu`
(branch `patches-r36.5`) against vanilla kernel 6.18.18 and achieved **BUILD EXIT CODE: 0**.

**Key discovery:** The OE4T driver already has ALL kernel compatibility code behind `#ifdef NV_*`
guards. These macros are normally auto-generated by the OE4T `nvconftest` detection system in the
umbrella build. When building standalone, none are defined — the fix is simply to pass them via
`KCPPFLAGS="-include /build/nv_compat.h"` with a force-include header.

**NV_* macros required for kernel 6.18:**

| Macro | Kernel version | API change |
|-------|---------------|------------|
| `NV_VM_AREA_STRUCT_HAS_CONST_VM_FLAGS` | 6.3+ | `vm_flags_set()`/`vm_flags_clear()` instead of `\|=`/`&=~` |
| `NV_CLASS_CREATE_HAS_NO_OWNER_ARG` | 6.4+ | `class_create(name)` — `THIS_MODULE` arg removed |
| `NV_CLASS_STRUCT_DEVNODE_HAS_CONST_DEV_ARG` | 6.2+ | `const struct device *` in devnode callback |
| `NV_FD_FILE_PRESENT` | 6.9+ | `fd_file(fd)` accessor instead of `fd.file` |
| `NV_FD_EMPTY_PRESENT` | 6.12+ | `fd_empty(fd)` instead of `!fd.file` |
| `NV_HRTIMER_SETUP_PRESENT` | 6.15+ | `hrtimer_setup()` merges `hrtimer_init` + `.function` |
| `NV_LINUX_IOSYS_MAP_H_PRESENT` | 5.18+ | `linux/iosys-map.h` (renamed from `dma-buf-map.h`) |
| `NV_PLATFORM_DRIVER_STRUCT_REMOVE_RETURNS_VOID` | 6.11+ | `platform_driver.remove` is `void(*)(...)` |

**NOT defined:** `NV_MODULE_IMPORT_NS_CALLS_STRINGIFY` — kernel 6.18 uses string literal form
`MODULE_IMPORT_NS("DMA_BUF")`, not the `STRINGIFY` macro form.

**L4T-only headers (stub the .c files — not covered by NV_* guards):**

| Header | Files stubbed | Reason |
|--------|---------------|--------|
| `soc/tegra/virt/hv-ivc.h` | `os/linux/nvgpu_ivm.c`, `os/linux/soc.c`, `common/cbc/contig_pool.c` | Tegra hypervisor IVC — unused on bare-metal |
| `linux/platform/tegra/mc_utils.h` | `os/linux/platform_ga10b_tegra.c` | Tegra memory controller BSP header |

> **Cleanup:** Delete the Job: `kubectl delete -f cluster/.tools/nvgpu-build-test.yaml`
> (no build cache to clean — using `emptyDir`, not `hostPath`)

### Phase 5 — NVIDIA container toolkit

> **This phase has been fully redesigned.** Phase 0 CUDA validation (see
> [jetson-cuda-extension.md](jetson-cuda-extension.md)) revealed that CDI-based device injection
> is the wrong approach for Jetson Tegra. The correct mechanism is
> `nvidia-container-runtime` with `libnvidia-container` compiled `--enable-tegra`, using CSV
> files to enumerate Tegra device nodes.
>
> **The detailed plan for this phase lives in
> [jetson-cuda-extension.md — Phase 1](jetson-cuda-extension.md).**
> Tracked under beads issue `home-ops-fwh`.

**Why CDI doesn't work for Jetson (summary):**
- `libcuda.so` from `nvidia-l4t-cuda` is a dGPU shim — it opens `/dev/nvidiactl` and
  `/dev/nvidia0` which do not exist on Tegra. Returns `CUDA_ERROR_SYSTEM_NOT_READY` (801).
- CDI injects device nodes but cannot substitute the wrong `libcuda.so` shim.
- The correct CUDA path on Tegra goes through `/dev/nvhost-ctrl-gpu`, `/dev/nvhost-gpu`, etc.
  via `libnvidia-container --enable-tegra` which reads CSV device lists.

**Confirmed Tegra device nodes** (from Phase 0 exploration of `nvgpu.ko`):

| Device | Purpose |
|--------|---------|
| `/dev/nvhost-ctrl-gpu` | GPU control |
| `/dev/nvhost-gpu` | GPU channel submission |
| `/dev/nvhost-as-gpu` | GPU address space |
| `/dev/nvhost-tsg-gpu` | Time-slice GPU |
| `/dev/nvmap` | NVIDIA memory allocator |
| (+ 9 more nvhost-* nodes) | See `l4t.csv` in jetson-cuda-extension.md |

### Phase 6 — RuntimeClass

Standard Talos NVIDIA flow creates a `RuntimeClass`:

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: nvidia
handler: nvidia
```

This is required on Jetson to select the `nvidia-container-runtime` handler for GPU workloads.
Pods wanting GPU access must set `runtimeClassName: nvidia`.

## Upstreaming to Mainline Siderolabs

The fork approach (`ghcr.io/mmalyska/nvgpu`) is a self-hosted workaround. Once the
driver builds cleanly and works on `nv1`, the natural next step is to contribute the extension
upstream so it appears in the [Talos Image Factory](https://factory.talos.dev/) catalogue.

### Prerequisites before opening PRs

1. The `nvgpu.ko` module must load successfully on a real Talos node (Phase 5–6 complete).
2. Both PRs must target the **same Talos kernel version** as the current `main` branch of
   each upstream repo. Check `Pkgfile` in `siderolabs/pkgs` for the active `kernel_version`.
3. Coordinate PR ordering: **pkgs PR must merge first** so the extensions PR can reference
   the published `ghcr.io/siderolabs/nvgpu-driver-pkg:<tag>` image.

## Gaps vs Standard Talos NVIDIA Flow

The [official Talos NVIDIA GPU guide](https://docs.siderolabs.com/talos/v1.12/configure-your-talos-cluster/hardware-and-drivers/nvidia-gpu)
covers discrete PCIe GPUs. Jetson requires additional steps:

| Gap | Standard flow | Jetson requirement |
|-----|--------------|-------------------|
| Kernel module | `nvidia.ko` loaded via extension | `nvgpu.ko` from OE4T (custom extension) |
| Module autoload | udev triggers load on `/dev/nvidia*` creation | Must declare `machine.kernel.modules: [{name: nvgpu}]` in talconfig.yaml |
| `bpf_jit_harden` sysctl | Set by toolkit installer | Must be explicitly set in `machine.sysctls` |
| Device nodes | `/dev/nvidia0`, `/dev/nvidiactl`, `/dev/nvidia-uvm` | `/dev/nvgpu`, `/dev/nvhost-*`, `/dev/nvmap` |
| Device plugin | `nvidia/k8s-device-plugin` finds `/dev/nvidia*` | Tegra device nodes differ — plugin may need CDI config or replacement |
| Container toolkit CDI | Auto-generated from NVML (`nvidia-ctk cdi generate`) | Must enumerate Tegra device paths manually |
| RuntimeClass | `nvidia` RuntimeClass required | Same — `nvidia` RuntimeClass still needed |

## Kernel API Compatibility Reference

The OE4T driver has all kernel API compatibility code in `#ifdef NV_*` guard blocks.
**No source patches are needed** — pass the right macros at compile time.

### Build flags for kernel 6.18

```bash
# Create nv_compat.h (force-included into every compilation unit)
printf '%s\n' \
  '#ifndef NV_COMPAT_H' \
  '#define NV_COMPAT_H' \
  '#define NV_VM_AREA_STRUCT_HAS_CONST_VM_FLAGS 1' \
  '#define NV_CLASS_CREATE_HAS_NO_OWNER_ARG 1' \
  '#define NV_CLASS_STRUCT_DEVNODE_HAS_CONST_DEV_ARG 1' \
  '#define NV_FD_FILE_PRESENT 1' \
  '#define NV_FD_EMPTY_PRESENT 1' \
  '#define NV_HRTIMER_SETUP_PRESENT 1' \
  '#define NV_LINUX_IOSYS_MAP_H_PRESENT 1' \
  '#define NV_PLATFORM_DRIVER_STRUCT_REMOVE_RETURNS_VOID 1' \
  '#endif' \
  > /tmp/nv_compat.h

# Build (KCPPFLAGS applies to all compilation units; KBUILD_MODPOST_WARN=1 skips
# symbol resolution check — undefined symbols resolve at runtime against the actual kernel)
make -j$(nproc) -C /path/to/kernel ARCH=arm64 \
  M=/path/to/nvgpu/drivers/gpu/nvgpu \
  KCPPFLAGS="-include /tmp/nv_compat.h" \
  KBUILD_MODPOST_WARN=1 \
  modules
```

### L4T-only file stubs (cannot be solved with macros)

Two headers are unconditionally included by platform-specific files and are not in vanilla kernel.
Replace those `.c` files with empty stubs — the code is only invoked under Tegra hypervisor:

```bash
# soc/tegra/virt/hv-ivc.h — Tegra hypervisor IVC
for f in $(grep -rl 'hv-ivc\.h' drivers/gpu/nvgpu/ | grep '\.c$'); do
  printf '/* stub: L4T-only header not available on vanilla kernel. */\n' > "$f"
done

# linux/platform/tegra/mc_utils.h — Tegra memory controller BSP
for f in $(grep -rl 'linux/platform/tegra/mc_utils\.h' drivers/gpu/nvgpu/ | grep '\.c$'); do
  printf '/* stub: L4T-only header not available on vanilla kernel. */\n' > "$f"
done
```

### Section mismatch warning (non-fatal)

The build produces one `modpost` warning:
```
WARNING: modpost: nvgpu: section mismatch in reference: gk20a_driver+0x8 (section: .data)
         -> gk20a_remove_wrapper (section: .exit.text)
```

This is a pre-existing issue in the driver — `gk20a_remove_wrapper` is marked `__exit` but
referenced from the non-exit `gk20a_driver` struct. It is a warning only and does not prevent
the module from loading.
